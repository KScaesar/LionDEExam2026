# **雄獅資訊-資料工程師-試題**

────────────────────────────────────────

## **重要：關於 AI 工具的使用**

很榮幸您參與雄獅資訊-資料工程師面試**，我們鼓勵、也期望您使用 AI 工具**（Claude、ChatGPT、Copilot、Cursor、Claude Code……任何您平常工作會用的都可以）。這份評量模擬的是本職位的真實工作方式：以 AI Coding Agent 為日常開發夥伴，**同時對 AI 產出保持批判性驗證**。

因此：

1. **使用 AI 完全不扣分**——我們期望的是您「駕馭 AI」的品質，寫出完美的答案。  
2. 如果有使用AI工具，必須繳交**與 AI 的完整協作紀錄**（對話匯出檔、session log 或 transcript）。  
3. 您必須繳交一頁 **「AI 協作報告」**，內容包含：

* 如何把任務拆解給 AI（附上關鍵 prompt 範例）  
* AI 在哪些地方出錯或不足？如何**發現**並糾正？  
* 哪些部分刻意不用 AI 或不採納 AI 的建議？為什麼？

4. **未繳交協作紀錄、明顯為AI產出**（例如產出遠超出紀錄所能解釋的範圍），視為重大扣分項。  
5. 提醒：題目的資料中埋有需要「親自檢視資料」才能發現的問題。直接把題目丟給 AI 一次生成的答案，在客觀評分點上會明顯失分——這是刻意的設計。

────────────────────────────────────────

## **環境需求**

* 任何熟悉的本地環境即可：**pandas / PySpark / DuckDB / SQL（任一）**，不需要雲端帳號。  
* 資料集隨題目附上（dataset/ 目錄，CSV 格式，約 11 萬筆訂單）。  
* 交付的程式碼需附說明文件（README；Word 或 Markdown 皆可）。

## **資料集說明**

| 檔案 | 內容 |
| :---- | :---- |
| orders\_base.csv | 訂單初始全量（2026-05-01 \~ 2026-06-30） |
| orders\_incremental\_day1\~3.csv | 三個每日增量批次（2026-07-01 \~ 07-03），含新訂單與舊訂單的狀態更新事件 |
| members.csv | 會員每日快照萃取（同一會員可能有多筆不同 extract\_date 的紀錄） |

*資料為合成資料，模擬旅遊業訂單場景。**資料品質狀況即為題目的一部分。***

## **Part A｜旅遊訂單資料倉建置（35 分，建議投入 4～5 小時）**

你是資料團隊的資深工程師，需要為訂單域建立分析用的資料倉儲。請完成：

### **A-1 星型模型設計與實作**

* 設計並實作：fact\_order（訂單事實表）、dim\_member（會員維度，**須以 SCD Type 2 保留歷史**）、dim\_product、dim\_date。  
* 以簡短文件說明：事實表**粒度**的決定、金額欄位的處理（幣別如何統一）、維度鍵的設計。

### **A-2 增量、冪等、可重跑的 ETL 管線**

* 實作管線：先載入 orders\_base，再依序處理三個每日增量檔。  
* 管線必須**冪等**：同一個增量檔重複執行兩次，事實表結果必須一致。**請在 README 中展示你的證明方式**（例如重跑前後的筆數與金額 checksum）。  
* 請說明你對「增量檔中出現的各種狀況」的處理策略。

### **A-3 資料品質檢核報告**

* 列出你在資料集中發現的**所有品質問題**：問題描述、影響筆數、偵測方式（附查詢）、你的處理決策與理由。  
* 格式不拘（Markdown 或 Word 表格皆可），這份報告是 Part A 的主要評分依據之一。

**Part A 繳交物：程式碼、模型設計說明、品質檢核報告、說明文件（含重跑證明）。文件類交付物 Word 或 Markdown 皆可。**

## **Part B｜AI 產出程式碼審查（15 分，建議投入 1～1.5 小時）**

隨題附上 buggy\_pipeline.py——這是一段「由 AI Coding Agent 生成」的 PySpark 訂單管線，宣稱已通過測試。你的主管請你在部署前做 code review。

8. 最後給出整體結論：這段程式碼可以部署嗎？若你是審查者，你會要求 AI（或工程師）如何修改？

## **Part C｜架構設計題（10 分，建議投入 1 小時）**

延續 Part A 的訂單域場景，公司決定將平台建置於 **Microsoft Fabric**，並要求資料平台同時支撐：

* 每日批次的 BI 報表（Power BI）  
* 訂單事件的**準即時**儀表板（分鐘級）  
* 面向 AI 應用的 **Text-to-SQL / RAG 問數服務**（自然語言查訂單數據）

請提出你的架構設計：

9. **架構圖**（Mermaid、draw.io 或手繪拍照皆可）＋各元件/分層的職責說明。  
10. 至少**兩個關鍵取捨**的論證，例如：Lakehouse vs Warehouse 的選擇、即時鏈路用 Eventstream 或微批次、語意層放在哪一層實作。請說明你的判斷依據，而不是只列出選項。  
11. 為了讓 Text-to-SQL 回答可信，資料平台需要準備哪些配套（語意層、元資料、權限、評估機制）？請具體到「你會建什麼東西」。  
12. **請明確標注**：這一題中哪些內容由 AI 協助產出、哪些是你自己的經驗判斷。

**Part C 繳交物**：設計文件（含架構圖）。

## **進階模組（D／E／F 三選一，必考，20 分，建議投入 2～3 小時）**

請依你的專長與職涯定位**選擇一個**模組作答，並在繳交物中**用 3～5 句話說明你為什麼選這個模組、而不選另外兩個**（這段說明本身也是評分參考）。

### **模組 D｜效能工程——把慢管線加速 10 倍**

module\_D/ 內含資料生成器與一段**正確但慢**的 baseline 管線：

13. 先執行 python generate\_perf\_data.py 在你的機器本地生成測試資料（約 600 個小檔、1,500 萬列、1.5GB，請預留磁碟空間）。  
14. 執行 python baseline\_etl.py，記下它的執行時間與 [CHECKSUM] 輸出——**baseline 的輸出就是正確答案**。  
15. **動手改之前**，先閱讀 baseline 並繳交一份「優化預測表」：你打算做哪幾項優化、每一項**預期**帶來幾倍加速、依據是什麼。  
16. 實作你的優化版本。引擎與工具不限（pandas 向量化、Polars、DuckDB、多進程、Spark……都可以），唯一硬性要求：**輸出必須與 baseline 的 CHECKSUM 完全一致**。  
17. 在同一台機器上量測 before/after，報告總加速倍數、各項優化的實際貢獻，並**解釋每項優化的原理**（瓶頸原本是什麼：CPU、I/O 還是記憶體？）。

**目標**：總加速 ≥ 10 倍。**繳交物**：優化預測表、優化後程式碼、量測報告（含預測 vs 實測的差異分析）。

### **模組 E｜AI 問數的評分守門**

公司要上線 Text-to-SQL 問數服務，上線前需要一道**自動評分守門機制**。module\_E/ 內含：

* warehouse/：一組乾淨的迷你星型數倉（CSV）  
* questions.json：15 個業務問題＋**語意定義**（營收、淨營收等指標的權威定義）  
* llm\_answers.json：某 Text-to-SQL 服務產生的 **12 個回答**（SQL＋宣稱的結果＋解釋文字），品質不明

你的任務：

18. 建置自動守門機制，逐一判定 12 個回答 **PASS / FAIL＋理由**。建議包含規則層（SQL 可執行性、宣稱結果與實際執行結果比對）與 LLM-as-judge 層（語意正確性、解釋忠實度）；judge 的 prompt 與評分 rubric 是核心交付物。  
19. LLM 用什麼都可以（你自己的 ChatGPT／Claude 訂閱、API、甚至 chat 介面手動跑），但流程要可重複。  
20. 根據你抓到的錯誤型態，提出**語意層／元資料要補什麼**，才能讓這類錯誤在源頭根絕（具體到「要建什麼東西」）。

**評分方式**：你的判定會與標準答案比對（哪些回答有問題是預先設計的），**抓錯與誤殺都計分**。**繳交物**：守門程式／流程、12 個回答的判定表（PASS/FAIL＋理由）、judge prompt 與 rubric、語意層改善建議。

### **模組 F｜特徵管線除錯**

module\_F/churn\_training\_pipeline.py 是 DS 團隊的訂單取消預測管線（使用與 Part A 相同的資料集）：離線 AUC 好得驚人，上線兩週後幾乎等於亂猜，DS 確認過模型本身沒問題，請你診斷這條**特徵管線**。

21. **診斷報告**：找出所有導致離線指標虛高的問題，逐一說明洩漏機制（為什麼離線看起來很準、上線卻失效），以及你的驗證方法。  
22. **修正設計**：給出修正後的特徵管線設計（不必完整實作，虛擬碼＋文字說明即可），說明正確的特徵計算原則與資料切分方式。  
23. **防再犯機制**：如果要讓這類問題永遠進不了生產，你會在工程流程裡加什麼？

執行程式是選配（需要 scikit-learn），**純閱讀程式碼作答完全可以**。**繳交物**：診斷報告、修正設計、防再犯機制說明。
